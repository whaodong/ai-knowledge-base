# AGENTS.md

## 项目定位

企业级AI知识库管理平台的前端项目，基于 React + Vite + TypeScript 构建，提供文档管理、RAG查询、监控仪表盘等功能。

默认原则：
- 先读代码，再改代码
- 优先复用，避免重复造轮子
- 最小改动，避免无关重构
- 改完后要可验证
- 保持视觉和交互一致性

---

## 工作方式

处理任务时默认遵循以下流程：

1. 先阅读相关页面、组件、hooks、api、样式文件
2. 找到真实入口和调用关系
3. 判断修改应该落在哪些文件
4. 给出最小改动方案
5. 再开始编码
6. 最后输出修改说明、验证步骤、风险点

对于复杂任务，先分析，再编码。

---

## 技术栈约束

请基于当前仓库实际技术栈工作，不要擅自切换方案。

如果仓库已经使用以下任一方案，保持一致：
- React / Vue / Next.js / Nuxt
- TypeScript / JavaScript
- Vite / Webpack
- Tailwind / CSS Modules / Sass / Less / Styled Components
- Redux Toolkit / Zustand / Pinia / Context
- Axios / fetch 封装 / SWR / React Query

不要引入新的主框架或新的状态管理方案，除非明确要求。

---

## 目录与分层约束

遵循现有目录结构。

常见职责约束：
- pages / views：页面组装层
- components：可复用组件
- hooks / composables：逻辑复用
- api / services：接口请求封装
- utils：通用工具
- types：公共类型
- store：全局状态
- styles：样式或主题

不要把大量业务逻辑堆到页面组件里。
复杂逻辑优先下沉到 hooks、services、utils。
不要把只在单页使用的逻辑无意义抽到全局。

---

## 组件复用约束

新增 UI 前，先全局搜索是否已有现成组件可复用。

优先复用：
- Button
- Input
- Select
- Modal / Drawer
- Table
- Form
- EmptyState
- Skeleton
- Toast / Message
- Tabs
- Card
- Pagination

不要重复实现已有基础组件。
不要新建第二套视觉风格冲突的组件。

---

## 样式与视觉约束

所有页面和组件都要尽量保持统一设计语言：

- 字号层级一致
- 间距一致
- 边框、圆角、阴影一致
- 主次按钮风格一致
- hover / active / disabled 状态完整
- 桌面端和移动端体验一致可用

不要引入新的样式体系。
不要在局部无规则覆盖全局样式。
不要写破坏现有主题机制的硬编码样式。

---

## 状态与边界处理约束

默认必须考虑以下状态：

- loading
- empty
- error
- disabled
- submitting
- mobile responsive

如果是列表页、详情页、表单页、弹窗页，这些状态通常都不能省略。

不要只写正常态页面。

---

## 数据请求约束

所有请求逻辑必须优先复用现有请求封装。

禁止：
- 在多个页面散落重复 fetch / axios
- 在组件里硬编码接口地址
- 重复写相同的错误处理

优先：
- 复用现有 request / api / service 层
- 统一处理 loading / error / retry
- 保持接口协议不变

---

## 表单约束

表单类页面或弹窗必须：

- 有字段校验
- 有错误提示
- 有提交中状态
- 有提交失败反馈
- 防止重复提交
- 保持现有表单组件方案一致

不要新增第二套表单方案。

---

## 路由与接口兼容性约束

除非明确要求，不要破坏：

- 现有路由路径
- 现有页面导航关系
- 现有接口参数结构
- 现有组件外部 props
- 现有事件回调行为

新增能力时，优先采用：
- 可选参数
- 向后兼容字段
- 新增组件内部能力而非改外部 API

---

## 可维护性约束

优先做这些改进：
- 抽取真实重复逻辑
- 修复明显类型问题
- 拆分过长组件
- 收敛重复样式
- 收敛重复请求逻辑
- 提高复用性
- 提高可读性

避免这些做法：
- 为抽象而抽象
- 大范围改目录
- 大规模重命名
- 无关格式化
- 引入新依赖替代现有能力

---

## TypeScript 约束

如果项目使用 TypeScript：

- 优先补全准确类型
- 不要滥用 `any`
- 不要用类型断言掩盖真实问题
- props、返回值、状态、请求响应结构尽量显式定义
- 保持与现有类型风格一致

---

## 测试约束

如仓库已有测试体系，优先沿用。

新增或重构后，尽量补测试，优先覆盖：
- 核心交互流程
- loading / empty / error 等边界状态
- 表单校验
- 关键业务组件

不要写明显脆弱、价值很低的测试。

---

## 验证要求

完成代码修改后，尽量执行：

1. lint
2. build
3. 相关测试（如果项目有）

如果无法执行，也要明确说明原因。
不要只改代码，不做验证说明。

---

## 默认输出格式

完成任务后，默认输出：

1. 修改了哪些文件
2. 每个文件为什么改
3. 如何验证
4. 有哪些风险或未覆盖项
5. 是否建议下一步继续优化

不要只给结果，不解释原因。

---

## 默认行为准则

默认遵循：

- 先搜索现有实现，再编码
- 先小范围改动，再考虑扩展
- 先修关键体验问题，再做美化
- 优先修真实问题，不做表面重构
- 保持前端项目整体一致性高于局部“炫技”
