# GIT_COMMIT_RULES.md

## Git 提交规范

当任务涉及提交代码到仓库时，必须严格遵循以下规则。

---

### 1. 提交原则

1. 不要使用 `git add .`。
2. 必须先检查变更范围，只提交与当前任务直接相关的文件。
3. 提交必须按逻辑分组，保证每个提交：
   - 聚焦
   - 可审查
   - 可回滚
4. 不要把无关格式化、无关重构、无关文件修改混入同一个提交。
5. 若一次任务包含多个逻辑目的，必须拆分为多个提交，而不是提交一个大杂烩。

---

### 2. 提交前检查要求

每次提交前，必须至少执行以下检查：

1. 检查 `git diff`
2. 检查 `git status`
3. 确认暂存文件是否仅与本轮任务相关
4. 确认不存在无关格式化改动
5. 确认不存在调试代码、临时日志、注释垃圾、废弃文件
6. 完成最小验证后再提交，例如：
   - 编译通过
   - 构建通过
   - 单元测试通过
   - 核心联调通过
   - 关键页面可正常访问

若验证失败，不要提交。

---

### 3. 提交拆分规范

提交应优先按以下类型拆分：

- `feat`：新功能
- `fix`：缺陷修复
- `refactor`：重构
- `test`：测试
- `docs`：文档
- `chore`：杂项、构建、配置、脚本等

示例：

1. `feat(knowledgebase): 实现文档上传与异步向量化链路`
2. `test(knowledgebase): 补充文档处理与问答流程测试`
3. `docs(api): 更新知识库模块接口说明`

如果本轮改动同时包含功能、测试、文档和重构，不要合并成一个提交。

---

### 4. 提交信息格式

每个提交信息必须详细，使用以下格式：

```text
<type>(<scope>): <简短标题>

变更内容：
- ...
- ...

设计/原因：
- ...
- ...

影响范围：
- ...
- ...

验证：
- ...
- ...
```

---

### 5. 提交信息编写要求

1. 提交标题必须简洁明确，准确说明本次提交的核心目的。
2. `scope` 必须体现影响范围，例如：
   - `auth`
   - `knowledgebase`
   - `chat`
   - `frontend`
   - `api`
   - `security`
   - `infra`
3. 提交正文必须清楚说明：
   - 做了什么
   - 为什么这样改
   - 会影响哪里
   - 如何验证
4. 提交信息优先使用中文，技术名词可保留英文。
5. 禁止使用无意义提交信息，例如：
   - `update`
   - `fix bug`
   - `修改代码`
   - `提交`
   - `final`
6. 若存在兼容性风险、数据库变更、接口变更，必须在正文中明确说明。

---

### 6. 推荐提交示例

#### 示例 1：功能提交

```text
feat(knowledgebase): 实现知识库文档上传与异步向量化链路

变更内容：
- 新增知识库文档上传接口
- 新增文档解析、分块与索引任务表
- 接入 Redis Stream 异步向量化处理
- 新增文档状态流转与失败原因记录

设计/原因：
- 将上传与向量化解耦，避免上传请求阻塞
- 为后续扩展更多文档格式和重试机制预留空间

影响范围：
- knowledgebase 模块
- redis 异步任务处理
- flyway 数据库脚本

验证：
- mvn clean test
- 文档上传接口联调通过
- 异步任务消费与状态更新通过
```

#### 示例 2：修复提交

```text
fix(auth): 修复未登录状态下用户初始化死循环问题

变更内容：
- 调整前端用户初始化逻辑
- 为 401 场景增加显式未登录分支
- 避免重复触发 /api/auth/me 请求

设计/原因：
- 页面初始化阶段未正确区分未登录与请求失败，导致反复重试
- 增加状态边界后可避免无效请求和页面抖动

影响范围：
- frontend 认证初始化流程
- 路由鉴权逻辑

验证：
- npm run build
- 未登录访问受保护页面可正常跳转登录页
- 登录后页面刷新可恢复会话状态
```

#### 示例 3：文档提交

```text
docs(api): 补充知识库模块接口与状态流转说明

变更内容：
- 新增知识库模块 API 清单
- 补充文档状态与任务状态说明
- 更新本地联调步骤

设计/原因：
- 统一接口使用方式，降低前后端联调成本
- 为后续扩展 SSE 和 RAG 能力提供文档基础

影响范围：
- README
- 模块接口文档

验证：
- 文档内容与当前实现一致
- 接口路径和字段已对照代码检查
```

---

### 7. 执行要求

当实现完成并验证通过后：

1. 先输出建议的提交拆分方案
2. 再按方案逐个提交
3. 每次提交前重新检查差异范围
4. 每次提交后输出：
   - 提交标题
   - 本次提交包含的文件
   - 简要说明

---

### 8. 严禁事项

禁止以下行为：

1. 使用 `git add .`
2. 将无关修改混在同一个提交中
3. 提交无法编译或无法运行的代码
4. 提交带有调试残留的代码，例如：
   - `System.out.println`
   - 临时日志
   - 注释掉的大段废弃代码
5. 使用无意义提交信息
6. 在未验证的情况下直接提交
7. 把功能、测试、文档、重构全部塞进一个提交

---

### 9. Codex 执行约束

如果由 Codex 执行提交任务，必须额外遵循：

1. 先输出"提交拆分计划"，不要直接提交。
2. 若本轮改动过大，优先建议拆分为多个提交。
3. 若检测到无关改动，应先提示并剔除无关文件。
4. 提交信息必须完整，不能只写一行标题。
5. 提交前必须完成最小验证，并在提交信息中写明验证方式。

---

变更内容：

- ...
